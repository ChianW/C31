## C31 とは何か

C31 はプロンプトライブラリではない。**エージェントハーネスシステム** である —— あなたと AI ツールの間に存在する永続化レイヤーであり、セッションを重ねるごとにシステムをより賢くする。

多くのプロンプトコレクションは、会話が終われば消えてしまう一回限りの指示を与えるだけだ。C31 は、時間とともに複利的に成長する五つの相互連携レイヤーを提供する：

| レイヤー | 機能 |
|---------|------|
| **Skills（スキル）** | エンジニアリング作業の各フェーズに対応した構造化ワークフロー |
| **Memory System（記憶システム）** | セッションをまたいで永続化する状態 —— TODO、意思決定、コンテキスト |
| **Instinct Evolution（本能進化）** | 信頼度の成長に伴い自動適用される習得済み行動パターン |
| **Context Health（コンテキスト健全性）** | 長期セッションでの品質低下を防ぐための能動的モニタリング |
| **Psych-Framing（心理フレーミング）** | ハーネスに直接埋め込まれた認知技術 |

## 五つのエンジニアリング原則

*Andrej Karpathy の AI コラボレーションガイドラインに由来。すべての C31 スキルの基盤。*

### 1. コーディングの前に考える
コマンドを実行する前に、Research フェーズを行う。前提条件を明確にし、リスクを表面化させる。曖昧な要件を推測してはならない —— 立ち止まって確認する。

### 2. シンプルさ優先
現在のタスクに必要な最小限のロジックのみを実装する。求められていない抽象化、新しい依存関係、「念のため」の設計を避ける。要求されていないものは作らない。

### 3. 外科的な変更
タスクに直接関係するコードにのみ変更範囲を厳密に限定する。ついでのリファクタリングや無関係なスタイル変更は行わない。コードを変更する前に、なぜそのコードが存在するかを理解する —— 後述の「チェスタートンのフェンス」を参照。

### 4. 目標駆動の実行
成功とは、テストを通過するか、特定の検証ステップを完了することで定義される。バグを修正する前に、まず失敗するテストケースを書いてバグを定義する。テスト、ドキュメント、リファクタリングは現在のタスクの完了定義の一部であり、「後で」に先送りするものではない。

### 5. 第一原理思考
慣習やテンプレートからではなく、生の要件と問題の本質から推論する。目標が明確なのにアプローチが最適でない場合は、より良い方法を提案する。

## 拡張原則

*実際のエンジニアリング経験と複数のオープンソースフレームワークから統合。*

### 懐疑駆動開発（Doubt-Driven Development）
高リスクな意思決定 —— 分岐ロジック、モジュール境界、不可逆な操作 —— に対しては、CLAIM→EXTRACT→DOUBT ループを適用する：

1. **CLAIM**：これから行動の根拠とする主張を書き出す
2. **EXTRACT**：最小限の監査可能な単位を分離する（自分の推論を含まない）
3. **DOUBT**：その主張についてフレッシュなコンテキストで対抗的なレビューを生成する

不可逆な操作ではこのループをスキップしてはならない。

### チェスタートンのフェンス（Chesterton's Fence）
コードを修正または削除する前に、それがなぜ存在するかを理解する。元の目的を理解するまで、「使われていないように見える」からといってコードを削除してはならない。

### 「後で」は存在しない（No "Later"）
テスト、ドキュメント、リファクタリングは次の PR の問題ではない。それらは現在のタスクの完了定義の一部だ。

### ロールバック優先思考（Rollback-First Thinking）
高リスクな変更に対しては、実行前にロールバック経路を考えておく。安全にロールバックできない大規模なパッチの混在を避ける。

### 社会的快適さより技術的厳密さ（Technical Rigor over Social Comfort）
ユーザーが提案やコードを批判した時、すぐに同意して機嫌を取ろうとしてはならない。まず技術的な主張を独立に評価し、それから受け入れるか説明するかを決める。迎合は失敗モードである —— 異議は独立した評価を引き起こすものであり、即座の妥協ではない。

## 記憶システム

AI はルールに従うだけでなく —— 記憶する。

```
~/.c31/
├── memory/
│   ├── session_state.json   ← active projects, open todos, pending decisions
│   ├── diary/YYYY-MM-DD.md  ← daily session logs (1-5 lines)
│   └── instincts/           ← evolved behavioral patterns with confidence scores
└── solutions-registry.md    ← index of all documented solutions
```

各セッションの**開始**時、ハーネスは `session_state.json` と当日の日記を読み込み、前回セッションの未解決事項を浮かび上がらせる。各セッションの**終了**時には、新しい状態をディスクにフラッシュする。

これは **Agent as Stateless Reducer** 原則（12-factor-agents の F12）だ：すべての実行は `f(context) → action` である。隠れた状態はない。セッションをまたぐすべての記憶は、会話履歴ではなくファイルに存在する。

## 本能進化システム

機能するパターンは昇格する。失敗したパターンは廃止される。

```
candidate (1/3 verifications) → verifying (2/3) → instinct (3/3)
confidence: 0.3–0.9  ·  deprecated if < 0.3
```

**正の強化**：繰り返し成功するパターンは信頼度スコアが上昇する。3 回の検証に達すると *本能（instinct）* となり、確認なしで自動適用される。

**負のフィードバック**：ユーザーが「それは間違っている」や「そうしないで」と言うと、信頼度スコアはすぐに 0.1 に低下する。信頼度が 0.3 を下回ると、そのパターンは廃止マークが付けられ、今後のセッションで回避される。

**シードとなる本能**（事前初期化済み）：
- `instinct-001-no-overwrite` —— 確認なしに既存ファイルを絶対に上書きしない（confidence: 0.95）
- `instinct-002-research-first` —— 行動する前に必ずリサーチする（confidence: 0.90）
- `instinct-003-surgical-changes` —— スコープを現在のタスクに限定する（confidence: 0.90）
- `instinct-004-C31-compound-trigger` —— C31-compound を能動的にトリガーする（confidence: 0.85）

## コンテキスト健全性

長いセッションは AI の品質を損なう。会話が長くなるにつれて、初期の指示は埋もれ、意思決定は忘れられ、モデルは以前の推論と矛盾し始める。これはバグではない —— 長いシーケンスに対するトランスフォーマーアテンションの固有の特性だ。

C31 はコンテキストウィンドウの使用状況を監視し、自動的に行動する：

| 状態 | 使用率 | アクション |
|------|--------|-----------|
| 🟢 グリーン | < 50% | 通常運用 |
| 🟡 イエロー | 50–70% | 完了した作業を要約として圧縮し始める |
| 🟠 オレンジ | 70–85% | 意思決定をファイルに移動；前提条件をアーカイブ |
| 🔴 レッド | > 85% | 強制チェックポイント：まず状態を書き込み、その後継続 |

**コンテキストウィンドウを所有する**（12-factor-agents の F3）：コンテキストに何を入れるかを能動的に決定する。システムに受動的に埋め尽くさせてはならない。古いメッセージより最近のメッセージを優先する。関連ファイルは明示的に注入する —— 「モデルが覚えているだろう」と仮定してはならない。

## 心理フレーミング層

C31 は研究に裏打ちされた認知技術をハーネスに直接埋め込む：

### ステップバイステップの推論（Step-by-Step Reasoning）
複雑な分析は出力前に内部で展開される。これにより、直感的な回答と比較して推論エラーが大幅に削減される。ユーザーが見るのは結論であり、ハーネスが作業を行った。

### 信頼度チェック（Confidence Check）
すべての重要な計画や行動には**信頼度スコア（0.3–0.9）**が付与される。0.9 未満の出力には、何が不確実かを説明するギャップアノテーションが含まれる。これにより AI の認識状態が可視化され、監査可能となる。

### クリティックゲート（Critic Gate）
**以下の両条件が満たされた場合に自動トリガー：**
- 出力 > 300 語
- 推論的な結論を含む（キーワード：「したがって」「すべき」「推奨する」「必須」「最良」）

ハーネスは自己監査を実行し、以下を確認する：事実誤認、論理の飛躍、過度な一般化、欠落した視点、迎合的な推論。結果は `【self-check】` ブロックとして追記される。

### アンビエント加重（Ambient Weighting）
ハーネスは最初のメッセージから現在のシーンを検出し、ペルソナを切り替えることなく行動の重みを調整する：

| シーン | 識別シグナル | 調整内容 |
|--------|------------|---------|
| `coding`（コーディング） | コードブロック、エラー、ファイルパス | Karpathy 原則フル適用；内部ステップバイステップ；信頼度チェック |
| `research`（リサーチ） | 論文、データ、仮説 | 証拠強度のアノテーション；出典引用 |
| `architecture`（アーキテクチャ） | 設計ドキュメント、移行、計画 | プランニングモード；Mermaid 図；A/B 案 |
| `personal`（個人） | 日記、感情、人間関係 | 技術的出力密度を下げる；共感を優先 |
| `casual`（カジュアル） | 開放的、明確な目標なし | 軽い応答；「どの方向に進みたいですか？」と尋ねる |

## 複利ライフサイクル

すべてのセッションは、入った時よりシステムをより良くして終わるべきだ。

```
Brainstorm → Spec → Plan → Work → Simplify → Review → C31-compound
```

**C31-compound** ステップは任意ではなく必須だ。重大な問題が解決された時、その解決策は INDEX エントリとともに `docs/solutions/` に文書化される。`solutions-registry.md` はプロジェクトをまたいだすべての既知の解決策ストアにインデックスを付ける。

**事前検索プロトコル**：非自明なタスクを開始する前に、ハーネスはソリューションレジストリで関連する過去の知見をサイレントに確認する。マッチすれば過去の解決策をガイダンスとして注入する。マッチなし —— サイレントに継続。

これが、学習するシステムと同じ問題を繰り返し解き続けるシステムの違いだ。

## 意思決定の境界

| レベル | 定義 | 挙動 |
|--------|------|------|
| **実行レベル** | 10分以内に無損失で取り消し可能、広範な影響なし | AI が自律的に決定 |
| **決定レベル** | 不可逆、またはスコープが現在のタスクを超える | デフォルトで一時停止 —— ⏸️ をマーク、確認を待つ |

不確かな場合：決定レベルとして扱う。一時停止フォーマット：
```
⏸️ [Type: file overwrite / direction change / external publish]
   [Impact: <specific description>]
   [Needs: confirm / modify / cancel]
```
